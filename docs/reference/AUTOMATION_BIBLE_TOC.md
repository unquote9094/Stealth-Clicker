# 📘 [Blueprint] BookToKi Automation Bible (v2.0 대비)

> **"145번의 실패를 딛고, 146번째의 완경함을 만들다"**
> 이 목차는 단순한 매뉴얼이 아니다. 개발 철학이자 기술 전도 문서입니다.

---

## 🧠 Part 1. 자동화의 철학 (The Philosophy)
>
> *"왜 코딩보다 자동화에 집착해야 하는가"*

* **Ch 01. 145번의 교훈**: 수동 배포가 사람을 어떻게 망가뜨리나? (에러 사례 시리즈)
* **Ch 02. 게으름의 미학**: "같은 3번 반복하면 스크립트로 만든다"는 황금 룰.
* **Ch 03. 불안의 제거**: "또 실수하다 잘릴수있나?"라는 공포를 시스템으로 극복하기.
* **Ch 04. 자산화 (Assetization)**: 코드는 사라져도 `scripts` 폴더는 영원하다. (재사용성)

## 🏗️ Part 2. 아키텍처와 설계 (Architecture)
>
> *"견고한 시스템을 어떻게 만들었나"*

* **Ch 05. 3-Step Pipeline**: Start(준비) -> Safety(방어) -> Finish(종료)의 삼박자론.
* **Ch 06. Node.js as a Shell**: 왜 Bash가 아니라 Node.js인가? (크로스 플랫폼의 마력).
* **Ch 07. 파일 시스템의 지배자**: `fs` 모듈로 프로젝트 구조 장악하기.
* **Ch 08. 프로세스 오케스트레이터**: `child_process`로 외부 도구(Git, gh) 부리기.

## 🔧 Part 3. 도구 심화: Start & Finish (Tools Deep Dive)

* **Ch 09. `start-task.js` (1) - 무에서 유를 창조로**: 빈 폴더에서 프로젝트를 일으키는 마법.
* **Ch 10. `start-task.js` (2) - 이슈 트래커 자동화**: CLI로 GitHub을 조종하는 법.
* **Ch 11. `finish-task.js` (1) - 헌터-킬러 패턴**: 이슈를 생성하고(Hunter) 죽이는(Killer) 완전 순환.
* **Ch 12. `finish-task.js` (2) - 커밋 메시지 파서**: 커밋 로그를 의사결로로 만드는 컨벤션 규칙.

## 🛡️ Part 4. 안전장치: Safety Belt (Quality Assurance)

* **Ch 13. 방어적 프로그래밍**: "개발자는 언제는 실수한다"는 전제.
* **Ch 14. `test_sanity.js`의 진화**: 하드코딩에서 동적 로딩, 그리고 테스트 스위트까지.
* **Ch 15. CI/CD의 첫단추**: 로컬에서 망가 못하면 서버에서도 망가 못한다.
* **Ch 16. 실패 시나리오**: 스크립트가 실패했을 때의 복구 방법 (Rollback).

## 🚀 Part 5. 무한의 확장 (Expansion)

* **Ch 17. AI와의 협업 프로토콜**: `.gemini/rules.md`가 자동화에 미치는 영향.
* **Ch 18. Husky & Lint-staged**: 커밋하는 시간 개입하는 문지기들.
* **Ch 19. GitHub Actions**: 내 컴퓨터가 꺼져도 돌아가는 워크 자동화.
* **Ch 20. 결론**: v146에서 넘어 v200을 향한 로드맵.
